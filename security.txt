Here's the full picture, ranked by what would actually hurt you:

Critical — Fix Before Any Real Users
1. No rate limiting on auth endpoints
Login, register, and SSO have zero throttling. Anyone can brute-force passwords at thousands of attempts per second. One script can try every common password against every user account in minutes.
→ auth/backend/accounts/api_views.py:28,76,245

2. JWT WebSocket middleware uses wrong key
Both portals use SIGNING_KEY instead of VERIFYING_KEY in the WS middleware. Since portals don't have the private signing key, the token verification is effectively broken — someone can connect to any WebSocket channel without a valid token.
→ public/backend/public_platform/jwt_middleware.py:18 and internal/backend/internal_platform/jwt_middleware.py:18

3. DEBUG=True
Stack traces with full request context (headers, cookies, env vars) served to any user who triggers a 500 error. Immediate information disclosure.
→ All three .env files, line 2

4. No health check endpoints
No /health/ on any backend. Load balancers, Docker, and monitoring tools can't detect a dead service.

High — Will Cause Problems Under Real Load or Targeted Attack
5. Dev quick-login buttons unguarded
The Superadmin/Mentor/Student buttons with password='any' render in all environments — no import.meta.env.DEV check. Ships to production as-is.
→ public/frontend/src/pages/auth/LoginPage.tsx:16-20

6. File uploads accept any file type
17+ FileField instances with no extension or MIME validation. A user can upload .py, .sh, .exe. If a Django process ever imports from the media directory, or if the server misconfigures MIME types, this is remote code execution.
→ public/backend/core/models.py and internal/backend/org_portal/models.py — multiple lines

7. Shared cookie names across frontends
Both SPAs use gile_access / gile_refresh. Cookies on localhost are not port-scoped per RFC 6265 — the internal portal can read the public portal's token and vice versa. In a targeted attack, a compromised internal portal page can steal a public portal session.
→ Both apiClient.ts files, lines 30-39

8. N+1 queries on high-traffic views
The mentor/student browse page runs one DB query per user in a loop. With 50 mentors = 101 queries. Under 200 concurrent users hitting browse simultaneously, this becomes the primary DB bottleneck.
→ public/backend/core/api_views.py:277-315

Medium — Operational / Reliability
9. Silent exception swallowing in consumers
WebSocket consumers catch exceptions and silently return or pass with no logging. When a WS bug occurs in production you have zero visibility — no error, no trace, no indication anything went wrong.
→ public/backend/core/consumers.py:86,195 and internal/backend/org_portal/consumers.py

Architectural (Known, Carry Forward)
10. Accounts model triplication
Three identical accounts/models.py — one field change requires edits in 3 places + 3 separate migrations. Already documented, not dangerous today but accumulates tech debt.
