Critical Architectural Analysis
1. Media — The Biggest Ownership Problem
Both portals point to the exact same physical folder:


# both .env files:
MEDIA_ROOT=/Users/I770121/Documents/GILE org/media

media/
  profile_pics/        ← public portal writes here
  workspace_logos/     ← public portal
  workspace_covers/    ← public portal
  message_attachments/ ← public portal
  feed/                ← public portal
  task_documents/      ← public portal
  org_documents/       ← internal portal writes here
  org_dm_attachments/  ← internal portal
  training/            ← internal portal
  doc_templates/       ← internal portal
  form_responses/      ← internal portal
Right now it works accidentally because they use different subdirectory names. But:

Both backends serve /media/ — so http://localhost:8002/media/org_documents/x and http://localhost:8003/media/org_documents/x both resolve to the same file. No portal truly owns its media.
In production behind nginx, you'd have to decide which service handles /media/ — you can't have both.
No isolation: a bug in one service could overwrite or delete files belonging to the other.
Fix: Each portal needs its own MEDIA_ROOT:


public/backend/media/
internal/backend/media/


2. The Shadow Table Pattern — Clever but Fragile
The router comment explains the design:

"Shadow accounts tables on default so FK constraints resolve locally; runtime reads/writes go to auth_db via db_for_read/write above."

So gile_public and gile_internal each have a copy of the accounts tables that is always empty. PostgreSQL FK constraints point to the empty shadow, but actual data lives in gile_auth. This means:

select_related('user') and prefetch_related silently return nothing — they query gile_public.accounts_customuser which is empty
Any queryset that tries to JOIN across the DB boundary returns empty results with no error
allow_relation = True in the router suppresses Django's cross-DB warning, so you won't even see it
This is the highest-risk silent failure in the whole system. Every place that does workspace.mentor.user.username or assignment.student.user.email is potentially broken depending on how it's written.



3. Accounts Model Triplication
All three accounts/models.py are identical — 68 lines each. Adding one field to CustomUser requires:

Edit model in 3 places
Run makemigrations in 3 backends
Keep all 3 migration histories in sync
Update serializers in 3 places
There is no single source of truth for the user model. The auth service owns the data but doesn't own the schema definition — all three backends define it independently.



4. Token Logout Gap — 60-Minute Window
When a user logs out:

Auth service blacklists the refresh token in gile_auth
Public and internal backends never check the blacklist — they only verify the JWT signature using the public key
The access token remains valid on both portals for up to 60 minutes after logout
This is inherent to stateless JWT, but with RS256 and a shared public key the portals could call a /token/verify/ endpoint on the auth service, or the access token TTL should be dropped to 5–15 minutes.

5. Cookie Sharing Between Portals
Both frontends use identical cookie names gile_access / gile_refresh with path=/. Since both run on localhost (different ports, but cookies are not port-scoped by default), they share the same cookies:

Logging out on port 5173 deletes cookies → also logs out port 5174
Logging in on one portal overwrites the token for the other
This is partly intentional (SSO feel), but it means the portals are more coupled at the browser layer than they appear.

6. No STATIC_ROOT — Production Blocker
Neither public nor internal settings define STATIC_ROOT. Running collectstatic in production will fail. Django admin CSS/JS won't load in production without it.

7. SECRET_KEY — Identical Across All Three Services

# all three .env files:
SECRET_KEY=django-insecure-change-me-in-production-use-env-var
For JWT (RS256) this doesn't matter since keys are RSA. But Django uses SECRET_KEY for sessions, CSRF tokens, and signed cookies. Having the same key means a session cookie forged for one service is valid on all three. They should be different per service.

8. /Intern_Portal/ — Dead Weight With a Git Problem
It's an empty prototype with its own .git folder — which is what caused your git add . error at the start. It's not used anywhere, has no running server, and blocks the parent repo. It should be deleted or properly added as a submodule.

Summary Table
Issue	Severity	Impact
Shared MEDIA_ROOT	High	Both portals write to same disk location; production conflict
Shadow table / cross-DB select_related	High	Silent empty results on user-joined queries
Accounts model triplication	Medium	Schema drift risk; 3x maintenance cost
60-min logout gap	Medium	Stale tokens accepted after logout
Shared cookies on localhost	Medium	Portals not truly isolated at browser layer
No STATIC_ROOT	Medium	Production deployment will fail
Identical SECRET_KEY	Low-Med	Sessions not isolated between services

The system is well-structured for a dev project but has several things that would silently break or become security issues the moment it hits a real server. The media and shadow-table issues are the two that could cause real data bugs today.
